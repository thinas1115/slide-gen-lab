# AIエージェント共通ルール

このファイルは、リポジトリをcloneして作業するすべてのAIエージェント向けの公開ルールである。
OS、ユーザー名、clone先、ツールのインストール先、GitHub権限が異なる利用者を前提にする。

## 作業開始時に読むもの

1. [README.md](README.md)でセットアップと基本操作を確認する。
2. 内容データを扱う場合は[CONTENT_SCHEMA.md](CONTENT_SCHEMA.md)を確認する。
3. rendererやレイアウトを変更する場合は[docs/architecture.md](docs/architecture.md)と
   [EXTENDING.md](EXTENDING.md)を確認する。
4. 配色や既存デザインを変更する場合は[DESIGN_CUSTOMIZATION.md](DESIGN_CUSTOMIZATION.md)を確認する。

ローカル専用の`CLAUDE.md`や`.claude/`は公開ルールの正本ではなく、Gitの追跡対象にも含めない。

## 利用環境を決めつけない

- コマンドはリポジトリルートから実行できる相対パスで記載する。
- ユーザーのホーム、デスクトップ、ドキュメント、アプリケーション配置を推測しない。
- CLIやライブラリが特定の場所にあると仮定せず、PATHまたは実行環境の標準的な探索方法を使う。
- 必要なツールが見つからない場合は、利用者固有のパスをコードや文書へ書かず、前提条件として報告する。
- GitHub CLIの有無やリポジトリへの書き込み権限を仮定しない。利用できない場合は、通常のGit操作、
  Web UI、パッチ提示など、利用者が選べる代替手段を示す。

## 情報漏洩を防ぐ

- 認証情報、APIキー、Cookie、個人メール、ユーザー名、端末名、社内URLを追跡ファイルへ書かない。
- ローカルの絶対パス、一時ディレクトリ、インストール先をコード、コメント、Markdown、Issue、PRへ書かない。
- 入力ファイルの場所はCLI引数、環境変数、またはGit管理外の設定から受け取る。
- `out/`、ローカル設定、検証途中の出力、クリップボード画像をコミットしない。
- エラー出力やスクリーンショットを共有する前に、パス、ユーザー名、メール、トークンが含まれていないか確認する。

コミット前に、少なくとも次を確認する。検出結果にはOS標準パスやURLが含まれる場合があるため、
各行が公開可能かを人間またはAIが判断する。

```powershell
git diff --cached --check
git grep -n -I -E "[A-Za-z]:\\\\|/Users/|/home/|AppData|Program Files"
git status --short
```

## 記述言語

- 人が読む開発成果物と運用記録は原則として日本語で記述する。
- Issue、Pull Request、レビュー、コミット、進捗報告、完了報告、説明ドキュメントに英語の定型文を使わない。
- コード識別子、コマンド、正式な製品名、ログ原文など必要な英語はそのままでよいが、説明は日本語で書く。
- 利用者が別言語を明示した場合は、その指定を優先する。

## Git作業

- 既存の変更や未追跡ファイルを、利用者の許可なく削除・上書き・巻き戻ししない。
- 特定の著者名やメールアドレスを設定しない。cloneした利用者のGit設定を使用する。
- AI名の共同著者トレーラーは、利用者が明示的に求めた場合を除いて追加しない。
- リポジトリへ書き込み可能な場合は、Issue、作業ブランチ、品質検証、Pull Requestの順で変更を提出する。
- fork、読み取り専用clone、オフライン環境ではIssueやPull Requestを必須にせず、作業ブランチまたはパッチで変更を渡す。
- 公開後の履歴書き換えや強制pushは影響範囲を確認し、利用者の明示的な承認がある場合だけ実行する。

## 実装原則

- `content.json`には文言、type、要素間の関係などの構造だけを書き、座標・余白・フォント値を入れない。
- `slidegen/content*.py`と`slidegen/diagram_specs.py`は回帰試験用fixtureであり、新規資料の情報源・
  文言例として使わない。新規資料は利用者が示した要件と情報源だけから作る。
- 座標、文字実測、配線、描画順はrendererまたはレイアウトエンジンで決める。
- 新しいtypeを追加するときは、renderer、登録、validator、schema、AI向けtype一覧、ギャラリー、品質検証を同時に更新する。
- 新しいrendererは、標準配置、裁量余白の圧縮、ジャンル固有要素の縮小、`FitError`による明示停止を
  この順で実装する。提出品質を保つ最小値と、最小値でも収まらない過密入力のテストを必ず追加する。
- カードは独立項目の比較、選択、事例、KPIに使い、情報量が少ないという理由だけで反復パネルを並べない。
- AWSアイコンは無改変で扱い、追加素材の出典とライセンスを`slidegen/assets/CREDITS.md`へ記録する。

## 品質ゲート

コードや文書だけの変更は、対象テスト、リンク検査、`git diff --check`を実行する。
スライド出力に影響する変更は、次の順序で検証する。

1. `python slidegen/validate_content.py content.json`
2. 対象generatorでPPTXを`out/`へ生成する。
3. `python slidegen/check_layout.py <生成したPPTX>`を実行する。
4. PowerPointを利用できる場合は`render.ps1`でPNG化する。
5. 全ページを一覧と原寸の両方で目視確認する。

新しいrendererまたは収容処理を変更した場合は、通常入力の回帰テストに加え、余白圧縮、要素縮小、
最小値での明示停止までを通る過密入力テストを実行する。

`check_layout.py`の合格だけでは完成扱いにしない。文字の視認性、余白、コントラスト、線の見え方、
内容の正確性はPNGと情報源を使って確認する。
