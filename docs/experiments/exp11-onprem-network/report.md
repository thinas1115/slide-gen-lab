# 実証実験レポート: 第3のグリッド図解を仕様記述だけで書けるか (Issue #11)

実施日: 2026-07-13 / 実施者: Claude (このリポジトリの diagram_layout 改修に関与したAI。完全な初見ではない点は割り引いて読むこと)

## 題材と条件

- 本社オンプレNW構成図(インターネット境界 + 拠点VPN + サーバ室セグメント + 監視)。ノード8個・4列3行・入れ子コンテナ・同一列直結ペア2組・長距離の監視線
- 全ノード icon なし(box_node)。**assets/ 不要**
- 入力は [deck.json](deck.json)(content.json 形式、`generate_from_json.py` 経由)
- **エンジン(diagram_layout.py)は一切変更しない**。仕様修正のみで収束させる

## 試行ログ

| 試行 | 変更 | 結果 |
|---|---|---|
| 1 | 素直な仕様(via/exit/enter 全部自動) | **生成前エラー**: 縦不足(必須2.64in > 利用可能2.62in) |
| 2 | 監視サーバの sub を削除 | 生成成功・check_layout exit 0。**しかし目視で fw→core と mon→fw の完全レーン重複を発見**([証跡](try2-lane-overlap.png)) |
| 3 | fw→core を exit=bottom/enter=top に変更 | **生成前エラー**: MIN_SEG違反(0.03inの縮退区間×2)。エンジンの自己検証が捕捉 |
| 4 | 監視線を `outside_container(dc, left)` チャネル経由に変更 | **成功**([結果](result.png))。check_layout exit 0・目視OK |

## 記録事項(Issueの観点に対応)

**1. 座標や補正値を書きたくなった箇所**

**ゼロ。** 4試行を通して座標・サイズの数値は一度も書かなかった。修正はすべて既存語彙(sub削除 / exit・enter指定 / outside_containerチャネル)で足りた。SLOT_PITCH(FW左辺へのHTTPS+監視の2着信の自動分離)、DIRECT_GAP(ap→db・core→monの直結線の可視確保)、コンテナ境界免除(via明示時)はAWS図以外でもそのまま機能した。

**2. エラーメッセージだけで自己修正できたか**

- 試行3→4: **できた。** MIN_SEG違反のメッセージ(「via/exit/enterの組み合わせを見直して」)で方向転換できた
- 試行1→2: **半分。** エラーの提案は「行数を減らすかAREAを広げて」だが、(a) AREAはJSONに開かれていない語彙(座標なので意図的に非公開)、(b) 実際に効いた対処=subの削除はエラー文に含まれていない。行間計算がSUB_Hを織り込むことを知らない初見AIは詰まる可能性が高い → **改善提案①**
- 試行2の見た目問題(レーン重複): **エラーでは出ない。** check_layoutは線×線の重複を検知しない(既知の限界が実害として再現)。目視ゲートが必須である根拠がまた1つ増えた → **改善提案②**

**3. 提出品質か**

最終版は提出可能と判断。残る妥協点は監視レーンと拠点VPN線の直角交差1箇所(NW図の慣習として一般的な表現であり許容)。

## 結論と提案

**結論: グリッド図解ジャンル内では「座標ゼロ」は3枚目の題材・アイコン無し環境でも成立する。エンジン改修は一度も必要にならなかった。** レイヤーモデル(EXTENDING.md)の「L1はジャンル内では再利用できる」側の裏付けが取れた。

改善提案(いずれも小規模、別Issue推奨):

1. 縦不足エラーのメッセージに「subラベルの削除・行の統合」を対処として明記する(diagram_layout._auto_rows)。JSONユーザーが取れない「AREAを広げる」提案より先に出す
2. 自動Zルート同士の縦レーン重複(試行2の症状)を validate_edges 系の自己検証で検知する。同一x±εの縦区間が別エッジ間で重なったら警告

## 留保

実施者はエンジン改修者本人であり「初見のAIが書けるか」の検証としては弱い。CONTENT_SCHEMA.mdだけを渡した外部AI(Codex等)での再実験は、scratchpadの依頼プロンプト下書きを使って別途実施できる。
